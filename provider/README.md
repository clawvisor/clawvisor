# terraform-provider-clawvisor

A Terraform provider that configures a Clawvisor instance (OSS, instance-scoped)
or organization (Cloud, org-scoped via `org_id`) declaratively over its REST
API, using [terraform-plugin-framework]. It is a separate Go module
(`github.com/clawvisor/clawvisor/provider`) so the provider's dependencies never
enter the server's `go.mod`.

## v1 resources

Core (work against OSS + Cloud):

- `clawvisor_agent` — agent identity + bearer token (`cvis_`)
- `clawvisor_service_config` — per-service JSON config document
- `clawvisor_vault_entry` — push-mode vault secret (write-only server-side)
- `clawvisor_api_token` — long-lived scoped API token (`cvat_`)

Governance policies (require the `local_governance` capability — spec 06a on
OSS, or Clawvisor Cloud):

- `clawvisor_model_policy` — model allow/deny (singleton)
- `clawvisor_spend_cap` — per-window spend cap (daily/monthly)
- `clawvisor_content_policy` — content-scanning rule (block/flag)
- `clawvisor_task_policy` — task guidance (singleton)

The provider negotiates capabilities at startup via `GET /api/features` and
fails fast at plan/apply with an actionable error when a resource needs a
capability the target server does not report — never a silent no-op.

Cloud-only resources (`clawvisor_org`, `clawvisor_team`, `clawvisor_sso_config`,
…) and registry-mirror publishing are deferred to v1.1.

## Distribution (v1: monorepo path / local mirror)

Until the registry mirror ships (v1.1), install via a Terraform CLI dev
override. Build the binary and point Terraform at it:

```bash
cd provider && go build -o ~/.terraform.d/plugins/terraform-provider-clawvisor .
```

```hcl
# ~/.terraformrc
provider_installation {
  dev_overrides {
    "clawvisor/clawvisor" = "/absolute/path/to/your/GOBIN-or-plugins-dir"
  }
  direct {}
}
```

## Authentication & the two-phase bootstrap

The provider authenticates with a long-lived `cvat_` API token
(`CLAWVISOR_API_TOKEN` or the `api_token` attribute). On a fresh instance the
only credential is the deploy module's short-lived bootstrap token; see
`examples/bootstrap/main.tf` for the two-phase apply that mints a durable
instance-admin token and burns the bootstrap token.

All credential-bearing attributes are marked sensitive. Secrets
(`vault_entry.value`, `agent.token`, `api_token.token`) transit Terraform
state — use an encrypted state backend and, where supported, ephemeral /
write-only values.

## Testing

```bash
cd provider
go test ./internal/client/... ./internal/provider/...   # unit tests
TF_ACC=1 go test ./internal/provider/testacc/... -run TestAcc -v   # acceptance
```

The acceptance lane is hermetic: it boots a real `clawvisor-server` subprocess
and mints an instance-admin token via the 05-lite bootstrap flow, so it needs
no cloud credentials. The four governance-policy tests skip cleanly until spec
06a lands the `/api/governance/*` routes and the `local_governance` capability.
